# joycon-mapper

把 Switch Joy-Con 手柄按键映射成 macOS 快捷操作。**真实诉求**：语音编程时解放双手——摇杆控鼠标 + 按键 DIY（Fn 唤起麦克风、回车、删除等）。按键映射是刚需。

## 当前状态（✅ 全通）

**最终方案：pyjoycon（持续激活+读）→ 摇杆控鼠标 + 按键 DIY。** 不需要 IOKit/GameController/重启。

| 通路 | 状态 |
|------|------|
| pyjoycon 读 0x30 | ✅ 稳定，持续激活后按键/摇杆都上报 |
| 摇杆 (analog-sticks.right) | ✅ h 741~3461 中心~2101；v 783~2986 中心~1884 |
| 按键 (get_status buttons) | ✅ right.a/b/x/y、R、摇杆按下、Plus、Home 都能读 |
| 鼠标控制 (pynput) | ✅ 摇杆推动→光标移动（需辅助功能权限） |

**关键教训**：早期"按键读不到"是**手柄静默态**问题（不是不支持）。pyjoycon 初始化发激活 subcommand 并持续后台读，按键就正常上报。早期用裸 hid/IOKit 间歇读时手柄没被持续激活，才误判为不通。**用 pyjoycon 全程持有连接是关键。**

**注意**：单只 Joy-Con 横握时，相邻按键可能同时上报（如按 A 出 a+b），做映射时需观察处理。

**⚠️ Steam 冲突**：Steam 在后台运行时会接管手柄输入，导致按键时灵时不灵。用本工具前**关掉 Steam**（或在 Steam 设置里关掉控制器支持）。这是早期按键"间歇不上报"的真凶之一。

## 功能（已实现）

- 右摇杆 → 鼠标移动（已验证手感 OK）
- ZR 扳机 → 长按说话（hold Fn，唤起豆包语音输入，实测通）
- A=左键点击 / B=回车 / X=删除 / Y=右键
- 映射在 `joycon.py` 配置区，改一行即可换键

运行：`DYLD_LIBRARY_PATH=/opt/homebrew/Cellar/hidapi/0.15.0/lib .venv312/bin/python joycon.py`

- 点击型按键已加防抖（`DEBOUNCE=0.30s`，防单只横握抖动连发）；还连发就调大

## 产品定位（2026-06-27 确立，产品灵魂）

**一句话**：别人给你一个通用手柄映射工具，你给 AI Coding er 一套调好的、尽量脱离键盘的手柄工作流。

**与 JoyMapper 等通用工具最大区别**：它们是空白画布（自己慢慢配），本产品是**开箱即为 AI Coding 调好的方案**。

**核心目标**：AI Coding 时**尽可能少用键盘**。AI 帮你写代码本就少打字，本产品把剩下的"控制"也搬离键盘。

**铁律边界（砍复杂度的关键）**：**语音覆盖文字输入，手柄管控制**。
- 文字（说代码/说需求/改名）→ 全靠语音，不做手柄拼字母/虚拟键盘/手势输入（反人类，坚决不做）
- 控制（导航/快捷键/确认/光标）→ 手柄 + 触摸板协同
- 最后那点零星精确文本编辑 → 容许用键盘，不强求纯净

三大支柱：
1. **默认即最优（为 AI Coding 设计的键位）**：预设专为 AI Coding 调好，拿到就顺手。核心理念 = **语音为主 + 轻操作**。
2. **全握持/全组合通吃**：横握/竖握、只左手柄/只右手柄/双手柄，每种都有对应预设。
3. **仍可完全自定义**：预设是起点不是牢笼。

**AI Coding 默认预设核心动作**：
- 🎤 语音触发（最高频核心）— 说需求/代码（现：ZR 长按 hold Fn）
- ⏎ 确认/回车 — 提交对话
- 🖱️ 移光标 — 摇杆 / 触摸板（精确时用触摸板）
- ↕️ 上下翻看 — 滚动/方向，看 AI 输出
- ⎋ 取消/中断 — 停止生成
- ⌨️ 常用修饰组合 — Cmd+C/V/Z 等（脱键盘的关键补充）

**手柄+触摸板协同**：触摸板做精确光标/手势，手柄做高频快捷键+语音，互补。

## Backlog（待做，按用出痛点的优先级排）

记录于 2026-06-27，用户决定先用现有右手柄版本，攒真实痛点再做。核心方向是**从写死映射升级为可配置 profile 系统**（也是产品化的核心）。

1. **配置文件化**：把 `MAPPING`/摇杆参数抽到外部配置（TOML/JSON），改键位不用改代码。这是后续一切的地基。
2. **多手柄 + 多握持**：
   - 左手柄 vs 右手柄：按键位/摇杆方向不同，需各自的映射
   - 握持方式：竖握 vs 横握，同一只手柄键位语义不同（横握时 ABXY 排成方位可当方向键）
   - 需要按"哪只手柄 + 怎么握"自动或手动加载对应 profile
3. **协同模式**：和触摸板配合用——如触摸板移光标、手柄只做快捷键/语音；或手柄摇杆+触摸板各管一摊。
4. **模式切换**：一个键在"鼠标模式 / 方向键模式"等之间切换（摇杆按下 r-stick 实测读不到，需用别的键，如 ZL/Plus）。切换时给提示（声音/通知）知道当前模式。
5. **预设 + 完全自定义**：内置几套开箱即用预设（选场景就能用），同时允许完全自定义。
6. **方向键**：上下左右（切行/导航/选择）——和"鼠标移动"争用摇杆，靠模式切换分配。

待调手感（用户已反馈/可能要调）：摇杆速度 `MAX_SPEED`、死区 `DEADZONE`、点击防抖 `DEBOUNCE`。

## 关键协议事实

- Joy-Con 0x30 标准报告：byte[0]=0x30, byte[1]=timer(+3/帧), byte[2]=连接/电池, byte[3..5]=按键位, byte[6..11]=摇杆, byte[12..]=IMU
- 激活 subcommand（report 0x01）：`0x40 arg=0x01` Enable IMU；`0x03 arg=0x30` Set report mode 0x30
- rumble (report 0x10) 是唯一可靠的唤醒/激活手段
- 单只 Joy-Con 在 GameController 下是 `microGamepad`，双只组合才是 `extendedGamepad`
